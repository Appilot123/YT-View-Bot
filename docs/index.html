YT View Bot – Human-Like YouTube Watch Sessions

Run safe, human-like viewing sessions with duration control, quality/speed settings, proxy-per-session, and multi-account scaling.

Get Started on GitHub

Features

Duration Control

Set per-session watch time windows (min/max), randomize intervals, and simulate natural viewing patterns.

Proxy-Per-Session

Map unique residential/mobile proxies to each session with automatic rotation & health checks.

Quality & Speed

Control playback quality (e.g., 480p–1080p) and speed ranges to mimic realistic behavior.

Multi-Account Scaling

Run concurrent sessions across many accounts with queueing, retries, and resource limits.

Search-Based Discovery

Find videos via keyword search and navigate results before playback to reduce patterns.

Logs & Export

Structured logs with CSV/JSON export for auditing session ids, proxies, durations, and outcomes.

Use Cases & Benefits

Installation Guide

Set up YT View Bot locally. See full README on GitHub for advanced options.

Prerequisites

  • Node.js 18+ and npm (or pnpm)
  • Valid proxy list (HTTP(S)/SOCKS), one per line

Steps

  1. Clone and install:
# clone
git clone https://github.com/Appilot123/YT-View-Bot.git
cd YT-View-Bot

# install deps
npm install
# or: pnpm install
  1. Create a config:
# config.example.json -> config.json
cp config.example.json config.json

# edit config.json
# - targetUrls: array of YouTube URLs or search keywords
# - duration: { minSec: 90, maxSec: 320 }
# - quality: "720p"
# - speed: { min: 0.95, max: 1.15 }
# - concurrency: 4
# - proxiesFile: "./proxies.txt"
  1. Run sessions:
npm run start
# optional flags:
#   --dryRun       # log actions only
#   --headful      # show browser windows
#   --shuffle      # randomize order

FAQ

Is this safe and human-like?
The bot emphasizes human-like patterns (randomized durations, search navigation, varied quality/speed, staggered starts). Use unique, healthy proxies and conservative concurrency.
How should I manage proxies?
Use proxy-per-session with reputable residential/mobile providers. Rotate on failures, monitor latency, and avoid sharing IPs across many accounts.
How far can I scale?
Start small (1–3 concurrency), validate stability, then increase based on CPU/RAM/network and proxy pool size. Use the logs & CSV/JSON export to audit performance.
Where can I report issues or request features?
Use GitHub Issues and Discussions.
